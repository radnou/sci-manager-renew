# Audit Projet GererSCI (style conseil senior Big4)

Date: 2026-03-04  
Périmètre: backend FastAPI, frontend SvelteKit, migration Supabase, tests/CI

## 1) Synthèse exécutive

Le socle est prometteur mais encore en phase "prototype structuré", pas "production-ready".  
Le principal risque est de sécurité/gouvernance d'accès (authentification, RLS, usage de clé service) combiné à un décalage API/frontend et une qualité d'exécution CI incomplète.

Niveau global de maturité estimé:
- Produit/MVP: moyen
- Engineering delivery: moyen-faible
- Sécurité & conformité: faible
- Exploitabilité/ops: faible-moyen

## 2) Constats majeurs (priorisés)

### Critique (P1)

1. AuthZ/RLS incohérent: la policy RLS utilise `associes.id` au lieu d'un `user_id`, rendant le contrôle d'accès applicatif non fiable.
   - Référence: `supabase/migrations/001_init.sql:9-14`, `supabase/migrations/001_init.sql:37-42`

2. Le backend initialise Supabase avec une clé privilégiée (service role / fallback anon) sans séparation claire des contextes utilisateur.
   - Référence: `backend/app/core/config.py:34-35`, `backend/app/core/supabase_client.py:20-22`

3. Absence de couche auth côté API (pas de JWT validation/courant utilisateur) malgré opérations de données.
   - Référence: `backend/app/main.py:1-13`, `backend/app/api/v1/biens.py:1-18`, `backend/app/api/v1/loyers.py:1-17`

### Élevé (P2)

4. Incohérence contrat backend/frontend: `BienCreate` requiert `id_sci`, mais l'UI envoie seulement `adresse` (+ optionnel `ville`) => erreurs 422 probables en réel.
   - Référence: `backend/app/schemas/biens.py:18-20`, `frontend/src/routes/biens/+page.svelte:13-15`

5. Frontend typé "strict" non conforme: erreurs TS (`any` implicite) et warning a11y.
   - Référence: `frontend/src/lib/stripe.ts:3-8`, `frontend/src/routes/biens/+page.svelte:4`, `frontend/src/routes/dashboard/+page.svelte:4`, `frontend/src/routes/loyers/+page.svelte:4`, `frontend/src/lib/components/QuitusGenerator.svelte:12`

6. Tests frontend incomplets en CI locale: dépendance Playwright browser non provisionnée.
   - Référence: `frontend/vite.config.ts:21-26`

### Moyen (P3)

7. API frontend codée en chemins relatifs `/v1/*` sans proxy explicite Vite (risque d'échec en dev multi-ports).
   - Référence: `frontend/src/lib/api.ts:2`, `frontend/src/lib/api.ts:7`, `frontend/src/lib/api.ts:16`, `frontend/src/lib/api.ts:21`, `frontend/vite.config.ts:8-43`

8. Schémas Pydantic v2 utilisent des patterns dépréciés (`Config`, `orm_mode`, `.dict()`), générant des warnings.
   - Référence: `backend/app/schemas/biens.py:33-34`, `backend/app/schemas/loyers.py:23-24`, `backend/app/services/biens_service.py:19`, `backend/app/services/loyers_service.py:17`

9. Le test backend nécessite un `PYTHONPATH` explicite depuis `backend/`, sinon échec d'import.
   - Référence: `backend/tests/conftest.py:5`

## 3) Vérifications exécutées

- Backend:
  - `cd backend && pytest -q` => échec import (`ModuleNotFoundError: app`)
  - `cd backend && PYTHONPATH=. pytest -q` => 7 passed, warnings Pydantic v2

- Frontend:
  - `npm run check` => erreurs TypeScript + warning a11y
  - `npm run build` => succès
  - `npm run test:unit -- --run` et `npm run test:e2e` => échec Playwright (browser non installé)

## 4) Plan de remédiation 30/60/90 jours

### 0-30 jours (stabilisation)
- Corriger modèle d'identité DB (`associes.user_id`), réécrire policies RLS et couvrir toutes tables.
- Ajouter authentification API FastAPI (JWT Supabase + context user).
- Aligner contrats backend/frontend (`id_sci`, DTOs communs, gestion erreurs HTTP).
- Réparer baseline qualité: `npm run check` vert, tests frontend exécutables (installer browsers Playwright en CI).

### 31-60 jours (durcissement)
- Introduire observabilité minimum (logs structurés, trace IDs, erreurs centralisées).
- CORS strict + rate limiting + validations métier renforcées.
- Compléter tests d'intégration backend contre DB de test (en plus des mocks).

### 61-90 jours (scalabilité)
- Mettre en place pipeline CI/CD robuste (quality gates obligatoires).
- Séparer clairement environnements et secrets (dev/staging/prod).
- Préparer audit conformité opérationnelle (accès, secrets, runbooks, incidents).

## 6) Décision recommandée

Décision de pilotage: **geler la livraison de nouvelles features pendant 1 sprint** et exécuter un "hardening sprint" orienté sécurité + qualité d'exécution.  
Sans cette étape, le risque d'incident (données, auth, dette de livraison) est trop élevé pour une montée en charge sereine.
